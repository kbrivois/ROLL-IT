Version 2.0 (15-03-2013)

Le projet en est déjà à sa version 2.0. Cette nouvelle mouture du jeu embarque de nombreuses nouveautés depuis la première version.

  • En plus des murs normaux, le jeu est enrichi de murs rebondissants. Lorsque la bille en heurte un, elle est repoussée dans le sens inverse, à plus grande vitesse. Ils sont pour le moment dessinés en rouge à l'inverse des murs normaux en noir.
  • Pièges par excellence du jeu, les trous sont désormais fonctionnels. Si une bille vient à tomber dedans, la partie retourne à l'état initial.
  • Le principe des trappes est semblable à celui des trous. Si la bille tombe dedans, la partie s'arrête. A l'exception près que les trappes se ferment et s'ouvrent à intervalle régulier. L'utilisateur peut donc profiter du temps pendant lequel la trappe est fermée pour passer dessus.
  • Les shurikens, les célèbres étoiles de ninja, traverseront la zone de jeu de manière linéaire. Si par malheur un shuriken frappe la bille, le joueur doit recommencer le niveau du départ. C'est la dernière fonctionnalité sur laquelle nous avons travaillé. Elle nous a demandé beaucoup de temps pour des raisons de performance. En effet, les shurikens tournent sur eux-mêmes. Au début de notre développement, dans le cas où plusieurs shurikens étaient lancés en même temps, le jeu avait tendance à freezer sur certains smartphones. Ce problème a été résolu dans la version 2.0.
  • Les diamants disparaissent lorsque la bille les traverse. Il faut désormais que le trou de fin apparaisse uniquement quand tous les diamants ont disparu.
  • Pour ce qui est du trou de fin de partie, il est fonctionnel. Lorsque l'utilisateur rentre dans le trou de fin, la partie s'arrête et l'utilisateur est félicité pour son adresse. Pour le moment, le trou de fin est matérialisé par une croix rouge.
  • Nous avons amélioré la gestion des API pour l'orientation et l'accéléromètre suivant les smartphones.
  • Il est désormais possible de changer la langue du jeu. Dans cette version, 4 langues sont disponibles : français, anglais, espagnol et portugais. D'autres langues seront ajoutées dans la prochaine version.
  • En ce qui concerne le stockage des données du jeu, nous avons opté pour l'API Local Storage. Nos premiers tests ont été concluants. Le lien entre l'interface et les données sera présent dans la version ultérieure.
  • Le fonctionnement des menus est terminé. L'utilisateur peut naviguer dans le jeu mais ne peut pas encore lancer de nouvelle partie. Il manque le lien avec les données.

Le jeu peut être testé sur la branch gh-pages dans le dossier jeu :
http://kbrivois.github.com/ROLL-IT/jeu

Le projet vit toujours (11-03-2013)

Il y a eu peu d'activité sur le projet ces derniers temps. En effet, les membres du projet avaient d'autres échéances. Désormais, nous avons plus de temps pour avancer sur ROLL IT!

  • Nous nous sommes replongés dans l'API Device Motion et Device Orientation pour que le jeu soit compatible avec le plus grand nombre de smartphones. Sur un téléphone équipé de FirefoxOS, c'est l'API Device Motion qui fonctionnera.
  • Lors d'une séance projet DLL, nous avons eu l'occasion de tester notre application sur un smartphone équipé du système d'exploitation FirefoxOS. Nous nous sommes focalisés sur des tests de performance, et nos tests ont été concluants. Le jeu est fluide.
  • Nos efforts se portent désormais sur le stockage des données. Nous avons décidé d'utiliser l'API Local Storage. Bien que cette API soit synchrone, cela ne pose pas de problèmes de conception pour le jeu, car l'API sera sollicitée uniquement en début et en fin de partie.
  • L'internationalisation est terminée. Une nouvelle classe a été créée pour contenir l'ensemble des traductions (Langues.js). Pour le moment, le jeu est disponible dans les langues Française, Anglaise, Portugaise et Espagnole.

Point sur le projet (13-02-2013)

Le projet se dirige tranquillement vers la version 0.2. Les commits sont plus rares en ce moment. En effet, d'autres échéances arrivent à grands pas, notamment le projet R&D. Il est à rendre pour le 24 février. C'est pourquoi les membres du projet mettent les bouchées doubles pour terminer ce travail dans leur groupe respectif.

Cependant, le projet ROLL IT! continue d'avancer.

  • Une nouvelle branche a été créée ; gh-pages. Cette nouvelle branche nous permettra de bénéficier d'un serveur HTTP, grâce aux services de GitHub. Ainsi, le wiki va être dupliqué sur la page d'accueil du serveur et une version de production sera déposée dans le dossier /jeu/ du serveur. L'adresse du serveur est la suivante : http://kbrivois.github.com/ROLL-IT
  • Le mécanisme des menus se perfectionne de commit en commit. Dans l'ensemble, le fonctionnement de ces derniers est maintenant déterminé, reste à implémenter ce mécanisme dans les prochains menus qui n'ont pas encore vu le jour.
  • Une première version de l'icône pour FirefoxOS a été déposée sur le dépôt.
  • Au niveau du code, la déclaration des pièges et obstacles a été modifiée. Dorénavant, un tableau a été mis en place pour chaque élément de la zone de jeu. Nous avons pris cette décision pour plus de lisibilité et de maintenabilité.
  • En ce qui concerne le trou de fin, cette fonctionnalité a été développée et est dorénavant en ligne sur le dépôt.

Menu (06-02-2013)

Le temps passe et les issues se ferment ! Le projet continue à bien avancer. Une première version du menu du jeu devrait être mise en ligne dans les prochains jours. En ce qui concerne les pièges et les obstacles :

  • Le développement du fonctionnement des trous est terminé. Pour ce qui est du design de ces derniers, une première version est sur le point d'être validée par l'équipe du projet.
  • Autre piège, le développement des trappes est sur le point d'être achevé, mais le design n'a pas encore commencé.
  • Les murs rebondissants en sont au même point que les trappes.
  • Tandis que pour les shurikens, le design des étoiles de ninja est fait, mais le développement commence tout juste.

Pour rappel :

  • Piège par excellence pour une bille, le trou est aussi simple que redoutable. Si la bille tombe dedans, le joueur recommence le niveau.
  • Les trappes sont similaires aux trous, à l'exception près qu'elles apparaissent et disparaissent à intervalle régulier à une position fixe.
  • Les shurikens, les célèbres étoiles de ninja, traverseront la zone de jeu de manière linéaire. Si par malheur un shuriken frappe la bille, le joueur doit recommencer le niveau.
  • Obstacles imprévisibles, les murs rebondissants seront sans nul doute les ennemis les plus redoutables. Si la bille heurte un mur rebondissant, elle est alors repoussée dans le sens inverse.

La structure du fichier où seront stockés les niveaux se dessine petit à petit. La pause en cours de partie est désormais complètement fonctionnelle.

Normes (04-02-2013)

Pour le bon déroulement du projet, nous nous sommes fixés quelques règles et normes pour coder. Ainsi, nous appliquons pour nos variables le principe de lowerCamelCase. Nos variables JavaScript sont également typées grâce à la première lettre. Par exemple, pour une variable integer : var iVariable = 12;

En ce qui concerne les commits, nous avons décidé de respecter la norme suivante :

  • [+] Une nouvelle fonctionnalité
  • [*] Une amélioration
  • [-] Un bug résolu
  • [~] Une fonctionnalité enlevée

Win overlay (02-02-2013)

Le projet avance. La version 0.1 finale est terminée et nous travaillons sur la prochaine version du jeu. Le système de collisions est sur le point d'être finalisé. Nous commençons à travailler sur le système de pièges et de trappes.

Étant donné que nous testons principalement sur des iPhones 3GS et donc Safari mobile, nous avons été surpris quand nous avons testé avec Firefox sur Android. Cette version du navigateur a un comportement similaire à celle présente sous FirefoxOS. Malheureusement, l'API Device Motion n'est pas supportée. C'est fort contraignant car elle marche très bien sur les versions mobiles de Safari. De ce fait, nous nous sommes attardés à gérer la compatibilité. Nous utiliserons donc en absence de l'API Device Motion, l'API Device Orientation. Elles sont similaires mais ne renvoient pas exactement les mêmes valeurs.

Test de l'interface (30-01-2013)

La version 0.1 finale n'est pas encore sortie, mais nous travaillons parallèlement déjà sur la 0.2. En ce qui concerne la version 0.1 finale, elle devrait sortir comme prévu initialement, pour le 4 février.

@kbrivois travaille sur le problème de rebond sur le bord inférieur et sur le bord droit. La détection des collisions sera son gros chantier pour la 0.2. Déjà mis en place par ses soins sur le projet Spacelash, c'est tout naturellement qu'il prend le lead sur le développement de cette fonctionnalité.

@PMarine a attaqué le développement d'une fonctionnalité indispensable du jeu prévue pour la 0.2, à savoir les menus. La sélection du niveau se fera en scroll horizontal dans un div. Un aperçu visuel des niveaux et leurs numéros permettront de les identifier.

@AAymeric a déjà mis en place une première version du chronomètre du jeu. La version finale de cette fonctionnalité sera présente dans la 0.2. La boule du jeu, élément graphique incontournable et centrale, en est à sa première version.

Pour tester, le jeu il faut impérativement avoir un smartphone supportant l'API Device Motion. Chaque personne du projet possède un iPhone 3GS. Les tests initiaux sont donc effectués sur ce type de smartphone.

Nous avons créé un environnement de test dédié pour chaque membre du projet sur un de nos serveurs webs. Ainsi, chacun peut effectuer ses propres tests, indépendamment des autres.
